# Quick Task 260712-n61: Close the PNG compression gap vs TinyPNG

## Problem

Flint's local PNG pipeline (hand-rolled median-cut quantizer + indexed-PNG encoder,
zero external deps for App Store sandbox safety) produced files noticeably larger than
pngquant/TinyPNG-class output. Diagnosis against real photographic + UI content
identified four independent gaps, ranked by measured byte impact:

1. **Fixed 256-color palette** (biggest) — always split to 256 boxes regardless of
   image complexity. A real photo that needs ~31 colors was shipped at 256.
   (`PNGColorQuantizer.swift` always ran to `cap = 256`.)
2. **No DEFLATE / filter optimization** — single weak `COMPRESSION_ZLIB` pass, PNG
   filter None on every row. Real encoders try all 5 row filters per scanline.
3. **Bit depth hardcoded to 8** — a ≤4-color image should be 2-bit (74% smaller on
   flat art). (`IndexedPNGEncoder.swift` wrote `ihdr.append(8)` unconditionally.)
4. **Alpha excluded from the quantizer's distance/MSE** — soft alpha edges destroyed
   when the palette shrank.

Not a factor (ruled out with numbers): dithering (Flint has none; dithering *increases*
size), metadata (already stripped by construction on the quantized path).

## Tasks

### Task 1 — Adaptive quality-target palette + alpha-aware MSE (`PNGColorQuantizer.swift`)

- **files:** `Tools/ImageCompress/PNGColorQuantizer.swift`
- **action:** Add `quantizeAdaptive(cgImage:quality:)` — a ladder search
  `[16,32,64,128,256]` that returns the smallest palette whose reconstruction MSE meets
  a quality-derived budget (maps the existing 0.0–1.0 quality slider to an RMS error
  budget). Compute exact reconstruction MSE **over R,G,B,A** inside `quantize` (one pass
  over unique colors, reusing the existing counts + lookup cache) and expose it as
  `QuantizedImage.mse`. Existing `quantize(cgImage:maxColors:)` signature unchanged.
- **verify:** `xcodebuild test -only-testing:FlintTests/PNGQuantizationTests` green;
  adaptive path picks fewer colors on photos, keeps 256 at quality 1.0.
- **done:** New static method compiles, MSE field populated, no public signature change.

### Task 2 — Adaptive bit depth + per-row filter selection (`IndexedPNGEncoder.swift`)

- **files:** `Tools/ImageCompress/IndexedPNGEncoder.swift`
- **action:** IHDR bit depth chosen from palette count (1/2/4/8) with matching pixel
  packing. Add per-row min-SAD PNG filter selection (None/Sub/Up/Average/Paeth). Because
  Apple's Compression-framework zlib is a weak matcher (filtering can inflate noisy data),
  **trial-deflate** both the all-None packed stream and the filtered stream and keep the
  smaller IDAT — guarantees filtering never makes a file bigger. `encode(...)` signature
  unchanged.
- **verify:** encoder tests green (signature, ImageIO validity, tRNS, round-trip,
  smaller-than-truecolor); `pngcheck` reports correct sub-8-bit depths.
- **done:** Valid PNGs at all depths; no image regresses vs the old None-only encoder.

### Task 3 — Wire quality into the PNG path (`ImageCompressTransformer.swift`)

- **files:** `Tools/ImageCompress/ImageCompressTransformer.swift`
- **action:** Thread the existing `quality` param into `writePNGCompressed` (private
  signature only) and call `quantizeAdaptive` instead of the fixed-256 `quantize`. The
  never-larger-than-original guard is retained.
- **verify:** `xcodebuild test -only-testing:FlintTests/ImageCompressTransformerTests` green.
- **done:** PNG path honors the quality slider; public `compress(url:quality:)` unchanged.

## Constraints honored

- No new dependencies (native-only; libimagequant/pngquant remain GPL/LGPL-blocked).
- No public API signature changes (`quantize`, `encode`, `compress`).
- Robustness preserved: all new paths nil-safe, cooperative cancellation intact.
