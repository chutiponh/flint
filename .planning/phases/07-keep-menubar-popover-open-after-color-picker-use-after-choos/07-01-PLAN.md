<objective>
After a color is picked via the eyedropper (`NSColorSampler`) or the system ColorPicker (`NSColorPanel`), the `.window`-style MenuBarExtra popover currently loses key focus and is force-closed by MenuBarExtraAccess's internal resign-key observer. This plan keeps the popover usable so the picked color lands in the Color tool and the user can copy any format and keep working ‚Äî for both pickers.

Purpose: Deliver the phase goal ‚Äî the picked color must never be lost, and the popover must survive both picker interactions (D-01), staying open live for the whole ColorPanel lifetime (D-04), while a normal no-picker dismiss remains unbroken.

Output: Two minimal source edits (eyedropper re-present + ColorPanel falling-edge watchdog) plus a manual UAT verification gate.
</objective>

<hard_constraints>
1. The "never-dismiss" mechanism (holding `isPopoverPresented = true` to suppress the close) is IMPOSSIBLE. MenuBarExtraAccess 1.3.0's internal `didResignKey` observer (MenuBarExtraAccess.swift lines 200-217) UNCONDITIONALLY calls `window.close()` and force-sets the binding to `false` whenever the popover window resigns key. The binding value is not consulted. The delivery mechanism for the phase intent IS the re-present (D-03), not binding suppression. Any change that assumes the binding alone prevents the close is wrong.
2. No new dependencies, no new files, no window subclassing/swizzling (ponytail / laziest-reliable-mechanism mode). Reuse the existing `clipboard.isPopoverPresented` lever only.
3. No automated tests are possible ‚Äî all 17 FlintTests target pure transformers/view-models; none touch NSWindow/key-status. Verification is MANUAL UAT only (`nyquist_validation: false` in config).
</hard_constraints>
