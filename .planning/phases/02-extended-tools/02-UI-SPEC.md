## Spacing Scale

The app uses a multiples-of-4 scale. Values below are the actual tokens established in Phase 1
(verified from JSONFormatterView, WarningBannerView, SyntaxEditorView source).

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4pt | Icon gaps, spacing between inline elements, HStack badge gaps |
| sm | 8pt | Compact padding (SyntaxEditorView inset, control bar vertical padding, row vertical padding) |
| md-h | 12pt | Horizontal control bar padding (verified: `.padding(.horizontal, 12)` in JSONFormatterView) |
| md | 16pt | Default SwiftUI `.padding()` equivalent; section separation |
| separator | 0pt | `Divider()` between major regions (no padding around dividers) |

All layout/padding tokens are multiples of 4.

### Corner Radii

Aesthetic radii вҖ” NOT layout spacing, exempt from the 4pt-multiple rule. Carried forward unchanged from verified Phase-1 source (WarningBannerView, InlineErrorView, JSONFormatterView).

| Token | Value | Usage |
|-------|-------|-------|
| corner-sm | 6pt | Small surface corner radius (WarningBannerView, InlineErrorView background, bit-buttons) |
| corner-md | 8pt | Control corner radius (Picker, Toggle, segment controls, swatch card) |

Exceptions:
- SyntaxEditorView text container inset: `NSSize(width: 8, height: 8)` (applied via NSTextView API, not SwiftUI padding).
- Popover total width: ~480pt (set by MenuBarExtra window; tools must be readable at this width вҖ” all layouts verified at 480pt).
- Main window minimum: 800 Г— 600pt (existing `MainWindowView` constraint; roomy layouts use this).
- Bit-field nibble gaps (Number Base): 4pt between bits within a nibble, 8pt between nibbles, 12pt between bytes вҖ” builder discretion within 4pt multiples.
- Touch/click targets for bit-toggle buttons: minimum 20 Г— 20pt (readability at 64-bit in 480pt).

## Color

This is a native macOS app. Colors are expressed as semantic NSColor/SwiftUI Color tokens, not
fixed hex values. The system resolves them correctly for Light mode, Dark mode, and all accent
color settings (INFRA-14 compliance).

### 60 / 30 / 10 Mapping to macOS Semantic Colors

| Role | Semantic Token | Usage |
|------|---------------|-------|
| Dominant (60%) вҖ” window/popover background | `Color(NSColor.windowBackgroundColor)` | Outermost popover and main window background |
| Secondary (30%) вҖ” card / control surfaces | `Color(NSColor.controlBackgroundColor)` | Control bars (indent picker row, flag toggles row, toolbar), panel headers, bit-field background, WCAG section card |
| Editor surface | `NSColor.textBackgroundColor` (AppKit, applied in `SyntaxEditorView.makeNSView`) | All `SyntaxEditorView` instances (Regex test string, Markdown editor, Diff Original/Changed editors) |
| Accent (10%) | `Color.accentColor` | Selection ring, active segment in segmented controls (Editor/Preview toggle, unified/side-by-side toggle), hover state `.accentColor.opacity(0.08)`, capture-group-0 highlight base |
| Separator | `Color(NSColor.separatorColor)` | `Divider()` between control bar and editor region; between editor and output |
| Destructive | `Color.red` | `WarningBannerView(.error)` icon and tint; overflow indicator badge tint; diff removed-line background |

Accent reserved for: active segment control state, keyboard-focused control ring, match/selection highlight base (Regex capture group 0 and diff current-difference navigator position). Do not apply accent to static labels or neutral surfaces.

### Per-Tool Color Additions

**Regex capture-group color coding (D-03):**
Assign one stable color per capture group index (0-based). Use a fixed palette of macOS-adaptive
colors (tested Light + Dark). Recommended palette (builder may substitute equivalent adaptive
colors):
- Group 0 (full match): `Color.accentColor.opacity(0.25)` background
- Group 1: `Color.yellow.opacity(0.30)`
- Group 2: `Color.green.opacity(0.25)`
- Group 3: `Color.purple.opacity(0.25)`
- Group 4: `Color.orange.opacity(0.25)`
- Groups 5+: cycle the palette

These are applied as `NSTextStorage` attribute background colors (not `.foregroundColor`) so text
remains readable in both Light and Dark mode.

**Text Diff line-level coding (D-16):**
- Added lines: `Color.green.opacity(0.18)` row background
- Removed lines: `Color.red.opacity(0.18)` row background
- Unchanged lines: no tint (inherits secondary surface)
- Word-level inline highlight within added text: `Color.green.opacity(0.40)`
- Word-level inline highlight within removed text: `Color.red.opacity(0.40)`

**Color Converter preview swatch (D-05):**
- Swatch is an `NSColor` (or `Color`) rendering of the current canonical RGBA value.
- Out-of-gamut OKLCH: swatch shows the gamut-clipped color; `WarningBannerView(.warning, "Out of sRGB gamut вҖ” clipped")` appears above swatch.

**WCAG contrast result badges:**
- Pass: `Color.green` icon/label
- Fail: `Color.red` icon/label
- Ratio value: `Color.primary` label, 13pt regular

**Interaction:**
- Pattern field: `TextField` or single-line `SyntaxEditorView`; monospaced 13pt; VoiceOver label "Regex pattern".
- Flag toggles: five `Toggle(.checkbox)` controls labeled "g (global)", "i (case insensitive)", "m (multiline)", "s (dot matches newline)", "x (verbose)" with `.accessibilityLabel` set on each. `g` controls enumerate-all vs first-match at the ViewModel level; it is NOT passed as `NSRegularExpression.Options`.
- "Patterns в–ҫ" dropdown: SwiftUI `Menu` button next to the pattern field; inserts preset pattern text into the pattern field on selection. Presets: Email, URL, Phone, Date (ISO 8601), IPv4 Address.
- Test-string editor: `SyntaxEditorView`, multi-line, monospaced 13pt. Highlight passes apply `NSTextStorage` attribute background colors per capture group (attribute-only, never reset `.string`). Re-entrancy guard required (Pitfall #5).
- Debounce + timeout (D-02, non-negotiable): 300ms debounce before launching background evaluation. Evaluation runs off `@MainActor` via `Task`. 2s timeout via `withThrowingTaskGroup`. On timeout: `WarningBannerView(.warning, "Pattern too slow вҖ” possible catastrophic backtracking")` shown; last-good highlight stays visible at `.opacity(0.4)` (outputDimmed pattern, CF-02). Cancel in-flight evaluation on each keystroke; check `Task.isCancelled` inside `NSRegularExpression.enumerateMatches` and set `stop.pointee = true`.
- Match results table: collapsible `DisclosureGroup`. Columns: Match # (1-indexed), Position (character offset), Full Match (truncated if >60 chars), Capture Groups (one column per group, named or numbered). Plain SwiftUI `List` is acceptable; a `Table` with named columns is preferred for the window layout.
- Replace mode: collapsible `DisclosureGroup`. Template `TextField`, monospaced. Preview renders in `CodeDisplayView` (read-only). CopyButtonView on preview output.
- `ToolShortcutActions` adopted: вҢҳвҮ§C copies current output (match count text + substitution preview if replace mode active); вҢҳвҢ« clears both pattern and test string.

**Detection predicate (CF-04):** Conservative вҖ” only match input that is obviously a regex literal in the form `/pattern/flags` (slash-delimited). Safer option: no detection predicate (search-only). Builder makes the final call and documents it in `RegexDefinition.make()`.

## Popover vs Window Duality

The recurring pattern for space-hungry tools (CF-05). Builder must detect context and switch layout.

| Tool | Popover (~480pt) | Main Window (вүҘ800pt) |
|------|-----------------|---------------------|
| Regex | Vertical stack (all sections) | Vertical stack with more vertical space; results table can show more rows |
| Color | Vertical stack (all sections) | Vertical stack; swatch can be taller; sliders more comfortable |
| Markdown | Segmented Editor/Preview toggle (one pane) | True side-by-side `HSplitView` (editor left, WKWebView right) |
| Number Base | Same layout (compact enough for both) | Same layout with more padding |
| Text Diff | Stacked editors + unified diff output | Stacked editors + side-by-side diff output (default) |

Detection pattern: query `NSWindow` frame width via `NSApp.keyWindow?.frame.width` or use a `GeometryReader`-based approach in SwiftUI. Threshold: < 600pt вҶ’ popover behavior; вүҘ 600pt вҶ’ window behavior. Builder discretion on exact threshold and implementation approach.

## Accessibility Contract

Required on ALL new interactive elements (INFRA-14, INFRA-15 вҖ” non-negotiable for all five tools).

| Element type | Requirement |
|-------------|-------------|
| Every `TextField` | `.accessibilityLabel("descriptive label")` вҖ” never rely on placeholder text alone |
| Every `Toggle` | `.accessibilityLabel(labelText)` |
| Every `Button` | `.accessibilityLabel(actionDescription)` |
| Every `Slider` | `.accessibilityLabel("channel name slider")` |
| Icon-only buttons (eyedropper, toolbar icons) | `.accessibilityLabel("human-readable action")` |
| `SyntaxEditorView` instances | `accessibilityLabel:` parameter in initializer (already wired) |
| `WarningBannerView` | `.accessibilityElement(children: .combine)` + `.accessibilityLabel(message)` (already implemented) |
| Capture-group color highlights | Color is supplementary вҖ” match-count badge and results table provide non-color path (color alone does not convey information) |
| Diff line-level color coding | Color is supplementary вҖ” `+`/`-`/` ` prefix characters provide non-color path |
| Bit-field buttons | `.accessibilityLabel("Bit [index], value [0 or 1]")` вҖ” state must be spoken |
| WCAG pass/fail badges | `.accessibilityLabel("AA Normal PASS/FAIL")` вҖ” not icon-only |

Dynamic Type: all `.font(.system(size:weight:design:))` usages automatically scale with Dynamic
Type. Do NOT use fixed pixel sizes via AppKit APIs for any text element that carries meaning.
The `.monospacedSystemFont(ofSize:13)` used in editors scales with `NSFont.preferredFont`.

VoiceOver navigation order: follow the visual top-to-bottom order of the VStack/HStack layout.
`accessibilityElement(children: .combine)` on composite rows (Color format rows, diff output rows).
