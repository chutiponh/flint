# Phase 4 Plan 6: Stable Signing Identity for Paste-Back (TCC) Fix

## Problem (root cause — verified)

Paste-back UAT (Tests 2–5) is blocked. The toggle-handler code in
`UI/PreferencesView.swift:228-267` is **correct**, but no Accessibility prompt
appears and no grant persists.

**Root cause:** This machine has **no code-signing certificate**
(`security find-identity -v -p codesigning` → "0 valid identities found"). So
`xcodebuild` falls back to **ad-hoc signing**, which gives the app **no stable
identity**:

- Yesterday's build: `Signature=adhoc,linker-signed`, `Identifier=Flint`, `CDHash=8933a224…`
- Today's rebuild:   `Signature=adhoc`,               `Identifier=com.flint.app`, `CDHash=206b34db…`

macOS TCC keys Accessibility grants on **(signing-identifier + cdhash + team)**.
Every rebuild mints a different ad-hoc identity, so:
- `AXIsProcessTrustedWithOptions(prompt:true)` won't reliably show the dialog for
  a fresh ad-hoc identity, and
- any grant that *is* made does not survive the next rebuild.

The "previous version worked" because an *earlier* build's identity (`Flint`,
cdhash `8933a224`) was already present in the Accessibility list. A rebuild
replaced it with a new stranger identity → it dropped out of the list.

**Fix:** Give the app a **stable signing identity** via a free **Apple
Development** certificate (Apple ID only, no paid program), and set a clean
reverse-DNS **bundle identifier `dev.chutipon.flint`**. Once the identity is
stable across rebuilds, TCC behaves correctly and paste-back is verifiable.

## Scope

In scope:
1. Rename bundle identifier `com.flint.app` → `dev.chutipon.flint` (app + tests).
2. Configure automatic signing with a free Apple Development cert (DEVELOPMENT_TEAM).
3. Harden the silent-failure UX: when no prompt is possible / not granted,
   surface immediate, clear feedback instead of a blank 30-second wait.
4. Re-run the blocked UAT checks (Tests 2–6).

Out of scope:
- Developer ID signing / notarization (already lives in Phase 3 distribution;
  unaffected). Sparkle `SUPublicEDKey`/`SUFeedURL` placeholders untouched.

## Tasks

### Task 1 — Bundle identifier rename (mechanical)
**Files:** `Flint.xcodeproj/project.pbxproj`
- Replace all 4 `PRODUCT_BUNDLE_IDENTIFIER` values:
  - app target (Debug+Release): `com.flint.app` → `dev.chutipon.flint`
  - tests target (Debug+Release): `com.flint.app.tests` → `dev.chutipon.flint.tests`
- `Info.plist` `CFBundleIdentifier` already uses `$(PRODUCT_BUNDLE_IDENTIFIER)` —
  no change needed (verified). Nothing reads the bundle ID at runtime (verified).
**Verify:** `grep -c "com.flint" project.pbxproj` → 0.

### Task 2 — Signing setup (checkpoint:human — needs Apple ID in Xcode)
This step needs the user, once: Xcode → Settings → Accounts → add Apple ID.
That creates a free "Apple Development" cert + provisioning. Then either:
- In Xcode: target → Signing & Capabilities → check "Automatically manage
  signing" → select your Team. Xcode writes `DEVELOPMENT_TEAM = <TEAMID>` and
  `CODE_SIGN_STYLE = Automatic` into pbxproj, **or**
- Pass on the command line: `xcodebuild … DEVELOPMENT_TEAM=<TEAMID> \
  CODE_SIGN_STYLE=Automatic CODE_SIGN_IDENTITY="Apple Development"`.
**Verify:** after build, `codesign -dvvv Flint.app` shows
`Authority=Apple Development: <name>` and a non-empty `TeamIdentifier`
(NOT `adhoc`, NOT `TeamIdentifier=not set`).

### Task 3 — Harden silent-failure UX
**File:** `UI/PreferencesView.swift` (`handlePasteBackToggleOn`)
Current behavior: on no-grant, user waits 30s with zero feedback, then a generic
denial. Improvement (small, additive):
- Show an immediate "Waiting for Accessibility permission…" status as soon as the
  prompt is requested (so the 30s poll isn't a blank void).
- Keep the existing revert-on-timeout + orange denial + Open Settings button.
- ponytail: do NOT add ad-hoc-build *detection* logic — there's no clean public
  API to detect "this build can't be trusted by TCC"; the immediate-status +
  Open-Settings escape hatch already covers the confusing case. Add detection
  only if it proves insufficient after signing is in place.
**Verify:** toggling ON shows status text within ~1 frame, not after 30s.

## Risks / Notes
- Free Apple Development certs expire ~7 days for non-program accounts but
  auto-renew while signed into Xcode; fine for local verification. Developer ID
  (Phase 3) is the durable distribution path.
- Changing the bundle ID means the app re-registers as a *new* TCC identity once
  more (expected, one-time). After that, the cert keeps it stable.
- UserDefaults domain is keyed by bundle ID → existing prefs (e.g. hotkey,
  pasteBackEnabled) reset to defaults on first run under the new ID. Acceptable
  for a dev build; note it so it isn't mistaken for a regression.
