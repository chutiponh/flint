<objective>
Implement the optional paste-back feature (CONTEXT.md D-09) — the final, isolated piece of the keyboard loop. A default-OFF Preferences toggle enables a synthetic ⌘V that pastes the just-copied output row into the previously-focused app. This is the ONE sanctioned exception to CF-02's no-Accessibility-prompt guarantee: the permission is requested only on explicit opt-in (prompt-on-enable), the toggle reverts on denial, and permission is re-verified at paste time. RESEARCH OQ-01 confirmed definitively that no permission-free synthetic-paste path exists on macOS 14, so the default-off + opt-in design is mandatory.

Purpose: Phase 4 success criterion 5 (the keyboard loop ends in "copy or paste … without the hands leaving the keyboard") and requirement UX-04 (paste-back opt-in, INFRA-16). Completes the D-07/D-08/D-09 keyboard-loop feature begun in Plan 03.
Output: `PasteBackService.swift` (new), `pasteBackEnabled` pref, `previousFrontmostApp` capture in `HotkeyManager`, the `FlintApp` environment registration, the D-09 toggle + permission flow in `PreferencesView`, and the paste-back branch in the Color/Hash/NumberBase `.selectOutputRow` observers.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Flint -> macOS TCC (Accessibility) | Requesting and exercising Accessibility permission to post synthetic key events — a privilege escalation surface. |
| Flint -> previously-focused app (synthetic ⌘V) | A CGEvent ⌘V is injected into a different process's input stream. |
| stored pasteBackEnabled pref -> paste action | A persisted bool that, if trusted blindly, could paste after permission revocation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-11 | Elevation of privilege | Accessibility permission requested without user intent | mitigate (HIGH) | Permission is NEVER requested at startup or on hotkey use. `AXIsProcessTrustedWithOptions(prompt:true)` is called ONLY inside `handlePasteBackToggleOn()`, reached only by an explicit user toggle-ON. Default OFF (CF-02 sanctioned exception). Blocking human-verify checkpoint (Task 3) confirms no unprompted request. |
| T-04-12 | Elevation of privilege | Stored `pasteBackEnabled` honored after permission revoked | mitigate (HIGH) | `synthesizePaste` and every observer branch re-verify `AXIsProcessTrusted()` at paste time; on revocation the action degrades to copy-only silently. Stored bool is never the sole gate (RESEARCH Security Domain). Checkpoint step 5 verifies. |
| T-04-13 | Tampering | CGEvent ⌘V injected into the WRONG app | mitigate | Target app captured in `HotkeyManager` BEFORE the popover opens (Pitfall 2); `synthesizePaste(into:)` activates that specific captured `NSRunningApplication`; nil capture -> no paste. Only the just-copied output row text is on the clipboard — no unrelated clipboard hijack. |
| T-04-14 | Tampering | Synthetic paste injecting unintended clipboard contents | mitigate | The observer writes ONLY `outputForRow(index)`/`primaryOutput()` of the active tool to the pasteboard immediately before the synthetic ⌘V; clears prior contents first. No other data path writes the clipboard in this flow. |
| T-04-15 | Information disclosure | Persisting Accessibility status / focused-app identity | accept | `pasteBackEnabled` is a non-sensitive bool; `previousFrontmostApp` is transient in-memory, never written to history/disk (RESEARCH Security Domain). |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs — D-09 uses system frameworks (ApplicationServices, CoreGraphics, AppKit); no new SPM packages this phase (RESEARCH.md). Package Legitimacy Audit not applicable. |
</threat_model>
