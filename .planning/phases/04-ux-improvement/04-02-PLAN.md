<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Flint -> Sparkle feed (network) | Sparkle fetches the appcast over `SUFeedURL`; the response is untrusted until EdDSA-verified by Sparkle. |
| Sparkle delegate callbacks -> UI status string | Error messages from Sparkle (`localizedDescription`) are surfaced to the user. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03 | Tampering | Appcast / update bundle integrity | accept (delegated) | EdDSA signature verification is Sparkle's responsibility (`SUPublicEDKey`) and is NOT weakened here. This plan only reads delegate result callbacks; it does not bypass or alter Sparkle's verification (D-04). |
| T-04-04 | Denial of Service | Unreachable/placeholder localhost feed | mitigate | A failing/unreachable feed resolves via `didAbortWithError` to a clear `.error(message:)` with `error.localizedDescription` — never a silent failure or indefinite spinner (D-04, CF-01). `.checking` is left only when a terminal callback fires. |
| T-04-05 | Information disclosure | Update status persistence | mitigate | `updateStatus` is transient in-memory state, reset to `.idle` on next popover open per UI-SPEC; nothing written to history or disk. |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs — Sparkle 2.9.3 already integrated in Phase 3; no new SPM packages this phase (RESEARCH.md). Package Legitimacy Audit not applicable. |
</threat_model>
