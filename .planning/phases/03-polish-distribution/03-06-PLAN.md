<objective>
Make launcher detect()-routing pre-fill actually land for every DETECTABLE tool, not just Color.

Background (from gap diagnosis): the launcher drop path (MenuBarPopoverView.fileDrop and, after Plan 07, MainWindowView.fileDrop) routes a dropped/staged value by calling `toolRegistry.detect(from:)`, then `toolSeed.set(toolId:value:)`, then navigating to the matched tool. The matched tool is expected to read the staged value once via `toolSeed.consume(for:)` in `.onAppear`. Today ONLY `ColorView` does this (Tools/Color/ColorView.swift:35). The other detectable tools never read the seed, so launcher-routed pre-fill silently drops the value for them. This is the pre-existing half of UAT Test 8 ("detect() routes to the JWT Decoder pre-filled").

The detection chain (Core/Services/ToolRegistry.swift:51 `detect`) only returns tools that declare a `detectionPredicate`. Those are: json-formatter, base64, url-encoder, jwt-decoder, timestamp, uuid-generator (and color, already done). Tools with NO detection predicate are never returned by detect() and need NO seed-consume: the search-only tools (regex, markdown, number-base, text-diff) AND Hash — Hash declares a nil detection predicate (it is intentionally detect-excluded so arbitrary text never auto-routes to a hash), so despite being a non-search tool it is correctly NOT in the seed-consume set.

Purpose: this is wave-1 groundwork so that Plan 07's workspace launcher-routing drop produces a real pre-fill, satisfying the user decision's requirement that drops "load + route via the existing detect()/FileDropHandler logic."
Output: 6 tool views gain a one-shot seed-consume on appear, mirroring the proven ColorView pattern.
</objective>
