## Spacing Scale

Declared values — derived from existing codebase patterns (source: UI/Components/*.swift). All tokens are multiples of 4.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4pt | List row vertical insets (HistoryRowView), icon-to-label gaps |
| sm | 8pt | Banner inner padding vertical (DetectionBannerView), component gaps |
| lg | 16pt | Section spacing in onboarding window, intra-group element spacing |
| xl | 24pt | Onboarding window content padding, major block separation |
| 2xl | 32pt | Horizontal text padding in empty-state fallback view |
| 3xl | 48pt | Not commonly used — reserve for full-window centering whitespace |

Exceptions:
- Popover fixed size: 480×600pt (established INFRA-01, locked — do not change)
- Workspace window default: 900×650pt (INFRA-02 locked)
- Onboarding window: 480×360pt recommended — tall enough to show menubar icon callout + hotkey instruction + CTA without scroll
- Drag-over overlay: covers entire tool surface with 0pt inset (full-surface drop target per D-05)
- Corner radius: 6pt for banners/cards (WarningBannerView), 8pt for interactive hoverable surfaces (PinnedToolBarView)
- WarningBannerView vertical padding: 6pt (shipped Phase 1 component — measured from existing rendering, off-scale; locked to preserve shipped appearance, change deferred to a later polish pass)
- 12pt horizontal inset: used as banner inner horizontal padding, search bar horizontal inset, and list row horizontal insets across shipped Phase 1/2 components. While 12 is a multiple of 4, it falls outside the standard {4, 8, 16, 24, 32, 48, 64} scale. This value is a measured codebase value from shipped components; changing it would cause visual regression. Realignment to the nearest scale token (16pt) is deferred to a later polish pass.

## Typography

All sizes are `pt` in SwiftUI `.system(size:)`. Source: measured from existing UI/Components/*.swift. Declared type scale: 4 sizes maximum.

| Role | Size | Weight | Line Height / Notes |
|------|------|--------|-------------|
| Caption | 11pt | semibold (600) | HistoryRowView tool-name badge — `.font(.system(size: 11, weight: .semibold))` |
| Body | 13pt | regular (400) | Default for banner messages, list labels, tool labels, instruction paragraphs, services labels — `.font(.system(size: 13))` |
| Subhead | 15pt | regular (400) or semibold (600) | Search field text (`.font(.system(size: 15))`); onboarding section heads; drag-over overlay label (15pt semibold) |
| Display | 20pt | semibold (600) | Onboarding headline "Welcome to Flint" — `.font(.system(size: 20, weight: .semibold))` |

SwiftUI line height: defer to system defaults (macOS resolves automatically). No manual line-height override needed.

Code/monospaced variant: Body (13pt regular) with `.font(.system(size: 13, design: .monospaced))` for hash output and code display. This is a design variant of the Body tier, not a separate scale entry.

Exceptions (off-scale sizes inherited from shipped components — do not count toward the 4-size scale):
- 10pt: used in minor badge sub-labels in pre-existing Phase 1/2 components — locked to existing rendering, off-scale change deferred
- 12pt: used in WarningBannerView message text in pre-existing Phase 1 component — locked to existing rendering; collapses into Body (13pt) on next polish pass
- 22pt: SF Symbol glyph size for the pinned-icon in PinnedToolBarView — this is a system glyph render size, not a text style, and does not count toward the type scale

## Color

All colors use system semantic NSColor/SwiftUI Color tokens. Source: established in Phases 1–2, carried forward. No custom hex values.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `Color(NSColor.windowBackgroundColor)` | Popover background, onboarding window background, tool surface fill |
| Secondary (30%) | `Color(NSColor.controlBackgroundColor)` | Cards, detection banner background, code display background, history rows |
| Accent (10%) | `Color.accentColor` (system accent) | See reserved-for list below |
| Warning | `Color.yellow` at 0.15 opacity background | WarningBannerView .warning severity — JWT issues, drop-type mismatch |
| Destructive / Error | `Color.red` at 0.15 opacity background | WarningBannerView .error severity — rejected file type for text-only tools |
| Text primary | `Color.primary` / `NSColor.labelColor` | All body text, input text |
| Text secondary | `Color.secondary` | Placeholder text, timestamps, dismiss buttons, icon decorations |
| Separator | `Color(NSColor.separatorColor)` | Dividers between zones (search bar ↔ pinned bar ↔ body), banner borders |

Accent (`Color.accentColor`) reserved for:
- `.borderedProminent` button style (primary CTAs: "Open [Tool]", "Enable Launch at Login")
- Pinned tool hover background: `Color.accentColor.opacity(0.08)`
- Active/set bit in BitFieldView: `Color.accentColor.opacity(0.15)` background, `Color.accentColor` foreground
- Progress indicator in file-hash progress view
- Drag-over overlay border/highlight when a valid drop is detected

Accent is NOT used for: passive text, secondary labels, icons, borders, backgrounds.

Phase 3 new semantic:
- Drag-over overlay fill: `Color.accentColor.opacity(0.08)` — same hover shade already in PinnedToolBarView
- Drag-over overlay border: `Color.accentColor` at full opacity, 2pt stroke, cornerRadius 8pt
- Drag rejected fill: `Color.red.opacity(0.08)` — mirrors WarningBannerView .error background tone

## Interaction Contracts

### Services Menu Integration (DIST-01, D-01 / D-02 / D-03)

- One Services entry "Open in Flint" — no per-tool entries.
- On invoke with matched content: skip detection banner, immediately open matched tool pre-filled via `ToolSeed`. No confirm step.
- On invoke with no match: open launcher (popover or workspace) with the selected text staged in the search/input area. User selects tool manually. No error state — never a dead end.
- Activation dance: same `WindowCoordinator` pattern as preferences/workspace — `.accessory` → `.regular` → activate → restore on close.

### Drag-and-Drop (DIST-02, D-04 / D-05 / D-06)

- Drop onto open tool: loads into that tool directly. Does not reroute.
- Drop onto launcher: reads text content, runs `detect()`, opens best-matched tool pre-filled.
- Drop overlay: appears on drag enter, disappears on drag exit or drop completion. Covers the entire tool/launcher surface. No permanent drop zone widget.
- Overlay visual: semi-transparent `Color.accentColor.opacity(0.08)` fill + `Color.accentColor` 2pt border + "Drop to load" label centered at 15pt semibold weight.
- Validation feedback: overlay turns red (`Color.red.opacity(0.08)` + `Color.red` border) if dragged item is rejected type. Label updates to rejection copy. `WarningBannerView` appears after drop completes with explanation.
- Binary files in Base64/Hash: accepted, processed off-main via existing chunked pipeline. Progress uses `ProgressHashView` (existing component).
- Text-only tools receiving binary: reject at drop, show WarningBannerView .warning.
- Animation: overlay fades in/out with `.easeOut(duration: 0.15)` — matches existing DetectionBannerView transition duration.

### Onboarding Window (DIST-03, D-07)

- Shown once on first run (gated by a boolean in `PreferencesStore` or `UserDefaults`).
- Single non-carousel window, not a sheet. Uses `WindowCoordinator` activation dance (opens above frontmost app).
- Window size: 480×360pt. Not resizable.
- Two steps shown inline (no carousel, no pagination dots): Step 1 (menubar callout), Step 2 (hotkey).
- Primary CTA ("Enable Launch at Login") calls `SMAppService.mainApp.register()` via existing `PreferencesStore.launchAtLogin` toggle — no new code path.
- "Skip" / "Get Started" dismisses window and sets the first-run flag — never shown again.
- No animation between steps (it is a single-screen layout, not a wizard).
- VoiceOver: all text readable in logical order — headline → body → CTA → skip.

### Sparkle Auto-Update (DIST-04, D-08 / D-09)

- No custom Flint UI for the update flow — Sparkle provides its own standard macOS update sheet.
- Flint owns: embedding EdDSA public key in `Info.plist`, `SUFeedURL` in `Info.plist`, wiring `SPUStandardUpdaterController`.
- The updater controller is owned at `FlintApp` level alongside other services.
- Update check fires automatically on launch (Sparkle default). No manual "Check for Updates" button required in v1 (builder's call per Claude's Discretion).

## Accessibility Contract (INFRA-15 — carried from Phase 1, enforced in Phase 3)

- `OnboardingWindowView`: every Text, Button, and Image has a VoiceOver label or is `.accessibilityHidden(true)` if decorative.
- `DropOverlayView`: announces drop state via `.accessibilityLabel("Drop to load")` or `.accessibilityLabel("File type not accepted")`.
- `FlintServiceProvider`: no visible UI, no accessibility requirement.
- Services menu entry: macOS handles `NSServiceMenuEnabled` labeling automatically.
- All new interactive elements support Dynamic Type scaling (use `.font(.system(size:))` with no fixed-height containers).
- Light/Dark mode: all new components use system semantic colors (`NSColor.windowBackgroundColor`, `Color.accentColor`, `Color.primary`, `Color.secondary`) — automatic adaptation, no hardcoded hex.
