Implement the user-chosen fix (Option 2): make file drag-and-drop reachable by routing drops to the existing normal `WindowGroup(id: "workspace")` → `MainWindowView`, which (unlike the MenuBarExtra(.window) popover NSPanel) does NOT dismiss on resign-key. This is the single fix that unblocks UAT Tests 5, 6, 7, and 8 — all share the root cause that the popover tears down the moment the user clicks Finder to grab a file (Apple FB11984872; full diagnosis in .planning/debug/popover-dismiss-blocks-drag.md).

Key facts established during diagnosis (do NOT re-derive — they shape the minimal diff):
  - The drop LOGIC is already correct everywhere and survives in a normal window. Every tool view already carries its own `.fileDrop`/`.onDrop` + DropOverlayView writing directly to its ViewModel input (e.g. JSONFormatterView.swift:124, HashView.swift:38). So dropping on a SELECTED tool inside the workspace ALREADY works today for all 12 tools — Tests 5/6/7 mechanics need NO change. They only failed because the host popover vanished.
  - What is missing: (A) the workspace's launcher chrome (sidebar + no-tool detail) is not itself a drop target, so the "drop on launcher → detect() → route" behavior (UAT Test 8) has no home in the workspace; (B) there is no popover-independent, discoverable way to open the workspace for dragging.
  - Plan 06 (dependency) made detect()-routed pre-fill land for all detectable tools via ToolSeed consume.

This plan does NOT replace MenuBarExtra, does NOT touch MenuBarExtraAccess, and does NOT introduce a custom NSPanel (that was rejected Option 1). The popover stays exactly as-is for click/hotkey launch.

Purpose: turn the already-correct drop logic into an actually-reachable feature by giving it a stable host window and a discoverable entry point.
Output: MainWindowView becomes a launcher-routing drop target mirroring MenuBarPopoverView.fileDrop; the popover gains a discoverable "open workspace to drag files" affordance.
</objective>

|----------|-------------|
| Finder file → workspace drop | untrusted file contents (text or binary) cross into the app via .onDrop |
| dropped text → detect() → tool input | routed value flows into a tool ViewModel via ToolSeed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07-01 | Tampering | dropped file contents | accept | Same untrusted input tools already accept via paste; FileDropHandler decodes off-main and rejects non-UTF-8 / oversized post-drop (FileDropHandler.swift:46-67). Binary tools stream off-main with no size cap by design (D-06). No new attack surface vs. existing per-tool drops. |
| T-03-07-02 | Denial of service | very large dropped file on a text tool | mitigate | The reused fileDrop helper enforces the existing 5MB UTF-8 threshold and routes oversized files to onError → WarningBannerView (FileDropHandler.swift:46-55). Binary tools (Base64/Hash) intentionally uncapped but stream off-main, keeping the UI responsive (Test 7 verifies). |
| T-03-07-03 | Information disclosure | ToolSeed staged value | mitigate | consume(for:) is one-shot/clearing (ToolRegistry.swift:78-83); HMAC keys / JWT secrets are never seeded (View-local @State). |
| T-03-07-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs — pure Swift source edits reusing existing components. N/A. |
