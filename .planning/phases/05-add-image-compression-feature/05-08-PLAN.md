<objective>
Fix GAP 2 (UAT Test 6) per the LOCKED user decision (Option C, CONTEXT.md / 05-UAT.md):
compress-on-drop is immediate, so the quality slider can never affect the images already dropped.
Add an explicit "Re-compress at {n}%" button. Keep compress-on-drop. Show the button when rows are
non-empty AND quality changed since the last run; clicking re-runs compress() on the retained source
URLs at the new quality. Hidden/no-op when the batch is entirely lossless and only quality changed.

Locked constraints (do NOT deviate):
- Do NOT use Hash-style auto `.onChange(of: quality)` re-run — it would write a NEW -compressed-N file
  on every slider tick (D-07 disk side effect). Re-compression fires ONLY on explicit button press.
- Do NOT change the drop-driven trigger (05-UI-SPEC line 119: "Compress trigger is implicit (drop-driven)").

Purpose: Resolve the slider/workflow contradiction — give the user a clear, explicit affordance to
re-apply a changed quality to the existing batch, without spewing files.
Output: ViewModel recompress() + retained source URLs/last-run quality + the conditional View button.
</objective>
