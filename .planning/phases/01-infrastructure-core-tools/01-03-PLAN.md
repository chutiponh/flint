<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| clipboard/user → JWTTransformer | Untrusted token (and possibly crafted/malformed) crosses into decode + verify |
| user secret field → verify | Secret HMAC key entered by user; must stay in-memory only, never persisted |
| ViewModel → HistoryStore | Persisted row must contain token only, never the secret |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-ID | Information Disclosure | JWT HMAC secret written to SQLite history | mitigate | [BLOCKING] Secret is a View-local `@State`; `onSaveHistory` receives token only; verified by `sqlite3 SELECT input FROM historyEntry` never containing the secret (INFRA-09, pitfall #3). High severity — blocks completion. |
| T-03-T | Tampering | Signature verification | mitigate | HMAC verify uses CryptoKit constant-time `Data(mac) == sigData`; wrong secret → false (JWT-04). No custom crypto. |
| T-03-IV | Input Validation | base64url decode + malformed tokens | mitigate | `Data.fromBase64URL` handles `-`/`_`; 3-segment guard; pure `Result` decode; tested against malformed tokens (JWT-01, pitfall #4, INFRA-17). |
| T-03-R | Repudiation/Correctness | exp timezone bug | mitigate | `Date(timeIntervalSince1970:)` only (pitfall #11); regression-tested. |
| T-03-SP | Spoofing | alg:none acceptance | mitigate | alg:none surfaces a warning banner and is never treated as a valid signature (JWT-06). |
</threat_model>
