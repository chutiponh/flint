<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user input → Timestamp/Hash transformers | Untrusted numeric/text input crosses into transforms |
| file system → Hash file read | Arbitrary, possibly very large files read without sandbox |
| user HMAC key field → keyed hash | Secret key entered by user; in-memory only, never persisted |
| ViewModel → HistoryStore | Hash history row must contain input + hashes only, never the HMAC key |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-ID | Information Disclosure | Hash HMAC key written to SQLite history | mitigate | [BLOCKING] HMAC key is a View-local `@State`; `onSaveHistory` receives input + hashes only; verified by `sqlite3 SELECT input FROM historyEntry WHERE tool='hash'` never containing the key (INFRA-09, pitfall #3). High severity — blocks completion. |
| T-04-DOS | Denial of Service | Hash file OOM on large files | mitigate | Chunked 1MB `FileHandle` read in `Task.detached` with incremental hash contexts; never `Data(contentsOf:)` (pitfall #9, HASH-02). Verified with a 200MB file. |
| T-04-V | Cryptography correctness | Hash + HMAC primitives | mitigate | All hashing via system frameworks (CryptoKit/CommonCrypto/zlib); zero hand-rolled crypto (RESEARCH "Don't Hand-Roll"). |
| T-04-IV | Input Validation | Timestamp ambiguity + malformed input | mitigate | 11/12-digit inputs flagged `.ambiguous` with a visible selector (pitfall #8); pure `Result` transforms; no crash on bad input (INFRA-17). |
</threat_model>
